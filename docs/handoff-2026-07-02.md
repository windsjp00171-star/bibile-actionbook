## 專案是什麼

`bibile-actionbook` — 互動聖經閱讀網站(Flask + Supabase)。核心賣點是**context-aware 標註引擎**:經文裡的人名/地名/族名可以點,跳卡片看背景、地圖、世系圖。

**核心原則(必讀,所有判斷都以此為準)**:
- 標註是為了讓大家讀得順暢,不是全文注釋。只標「真正生僻、會卡住讀者」的專有名詞。
- **準確永遠優先於數量**。不確定的、位置不詳的族譜「誰生誰」,寧可跳過不標。
- 標註前**先看經卷定時代背景**(舊約猶大 ≠ 賣主的猶大),**再看上下文語境**(畫邊界→地名、數點兵→民族/支派、對人說話→人名),**再看前後字避免碎片誤標**(如「約書亞」不能被切成「書亞」)。

## 關鍵基礎設施

### `tools/bible_query.py` — 本地查詢工具(零 API 成本)
```bash
python tools/bible_query.py verse 約書亞記 15 3          # 看某節原文
python tools/bible_query.py entity 希伯崙                 # 查詞條目前設定
```

## 資料結構關鍵知識

### `data/entities.json`
- 單一詞條：`{"type":"person|place|concept", "name":..., "name_en":..., "testament":"OT|NT|both", "desc":...}`
- 多義詞條（同名不同義，如「猶大」人/地/支派）：改成 **list**，`resolve_entity()` 四層優先度：書卷+章節 > 書卷 > 約別 > 通用。**list 內順序不影響優先度**，優先度由 books/chapters/testament 欄位決定。
- **常見陷阱**：新加的詞條如果沒設 `books`，會全域生效（好處：全域注入；風險：跟別的書卷同名詞條衝突）。這次任務中，**超過 10 次**是因為新地名跟已存在的人名同名（如「該隱」創世記人物 vs 約書亞記地名、「拿瑪」「基比亞」「希斯崙」「戶珥」「以利押」「示每」等），每次都要判斷是否需要 `books`/`chapters` 限定消歧義。**新增詞條前，務必用 `bible_query.py entity <name>` 先查有沒有同名已存在。**

### `mark_bible.py`
- `_SINGLE_CHAR_OK`：單字白名單（如 `噩`、`揝`、`圭`、`纛`）。單字預設不標，除非全經搜尋確認**唯一用法、零碰撞風險**才加白名單。加白名單前用 `bible_query.py dist <字>` + `search <字> --full` 確認。
- `_EXT_GUARD`：碎片守衛集合，防止長詞被短詞切碎（如「示羅密」被切成「示羅」+「密」、「摩西錄」被切成「摩西」+「錄」）。發現碎片誤標時加進這個 set。

### JSON 編輯注意事項（這次踩過的坑）
把單一詞條改成 list（做消歧義）時，**容易漏改或多改結尾的 `},`**，一定要編輯後立刻跑：
```bash
python3 -c "import json;json.load(open('data/entities.json'));print('JSON valid')"
```
再跑 `mark_bible.py` 的 resolve_entity 測試確認兩個語境都正確解析，不要只驗證 JSON 合法。

## 使用者的個性與期望（讀這段比讀程式碼更重要）

- 對 **token 成本很敏感**，目前沒有 Max 方案，會分次撥時間處理，不要預設他有無限預算。
- 喜歡**加減平衡**：加新功能前想清楚是否也該做減法（拿掉沒用的東西）。
- 回報時**已經自己做過上下文判斷**（先看經卷、再看語境），這些回報通常是對的，照著查驗即可，不用懷疑他抓的方向。
- 交接／snapshot 的時機由他主動提起（如「對話到了要做 SNAPSHOT」），要主動配合，不要等到硬性 context 限制才做。
- 不喜歡被一直問「要不要繼續」——一旦他說「你全做完」，就一路做到完，只在遇到需要他決策的模糊地帶才問。
- 支持信仰核心：這是為了讓人更順暢讀神的話，最終目標不是功能多，是**準確 + 好用**。
