## 專案是什麼

`bibile-actionbook` — 互動聖經閱讀網站(Flask + Supabase)。核心賣點是**context-aware 標註引擎**:經文裡的人名/地名/族名可以點,跳卡片看背景、地圖、世系圖。

**核心原則(必讀,所有判斷都以此為準)**:
- 標註是為了讓大家讀得順暢,不是全文注釋。只標「真正生僻、會卡住讀者」的專有名詞。
- **準確永遠優先於數量**。不確定的、位置不詳的族譜「誰生誰」,寧可跳過不標。
- 標註前**先看經卷定時代背景**(舊約猶大 ≠ 賣主的猶大),**再看上下文語境**(畫邊界→地名、數點兵→民族/支派、對人說話→人名),**再看前後字避免碎片誤標**(如「約書亞」不能被切成「書亞」)。

### 本輪新增的高效率作法:`diff.py` 逐節比對腳本
`bible_query.py mark --verses` 只印「本章標了什麼」,不夠看漏了什麼。本輪改用一個臨時腳本(存在 scratchpad,沒進 repo,下次要重寫或用類似邏輯):對整章每一節同時印出**原文**和**annotate() 後抓出的 data-entity 清單**,一次呼叫就能肉眼比對出哪節漏標、哪節標錯字。比單獨查 `mark`/`verse` 快很多,強烈建議下次比照重寫:

```python
import sys, os, json, re
ROOT = "/home/user/bibile-actionbook"
sys.path.insert(0, ROOT)
import mark_bible as mb

def show(book, ch):
    with open(os.path.join(ROOT, "cuv.json"), encoding="utf-8") as f:
        cuv = json.load(f)
    chap = cuv.get(book, {}).get(str(ch), {})
    print(f"=== {book} {ch} ===")
    for v in sorted(chap, key=int):
        text = chap[v]
        html = mb.annotate(text, book, ch)
        marked = re.findall(r'data-entity="[^"]*">([^<]*)</span>', html)
        print(f"{v}: {text}")
        print(f"   marked: {marked}")
# show(sys.argv[1], int(sys.argv[2]))
```

## 資料結構關鍵知識

### `data/entities.json`
- 單一詞條:`{"type":"person|place|concept", "name":..., "name_en":..., "testament":"OT|NT|both", "desc":...}`
- 多義詞條(同名不同義,如「猶大」人/地/支派):改成 **list**,`resolve_entity()` 四層優先度:書卷+章節 > 書卷 > 約別 > 通用。**list 內順序不影響優先度**,優先度由 books/chapters/testament 欄位決定。
- **常見陷阱**:新加的詞條如果沒設 `books`,會全域生效(好處:全域注入;風險:跟別的書卷同名詞條衝突)。本輪例子:「拉瑪」全經 43 次出現,士師記/撒母耳記上的「拉瑪」是便雅憫著名地標(撒母耳故鄉),但約書亞記19章同名出現 3 次卻是完全不同的小城(西緬/亞設/拿弗他利境內各一個,彼此也無法用章節分辨,故**刻意不標**,寧缺勿錯)。已將「拉瑪」加為 `books:["士師記"]` 限定詞條,**下次處理撒母耳記上時,直接把「撒母耳記上」加進這個既有詞條的 books 陣列即可**,不要重新開一條。
- **新增詞條前,務必用 `bible_query.py entity <name>` 先查有沒有同名已存在。**

### `mark_bible.py` 碎片誤標的正確處理方式(本輪重要發現,務必先讀)
`_ALL_NAMES` 是把所有詞條**按長度由長到短排序**後組成的 regex(`_ALL_RE`),Python 的 `|` 交替匹配在同一起始位置會依序嘗試、取第一個成功的——因為長詞排在前面,**只要把完整長地名／人名登記為詞條,regex 就會自動優先匹配長詞,不需要額外去改 `_EXT_GUARD`**。
- 本輪修好的碎片誤標範例:「西西拉」(士師記重要角色,底波拉故事的敵將)長期被誤標成無關的「西拉」;「瑪挪亞」(參孫父親)被誤標成「挪亞」;「基列西弗」被切成「基列」+「西弗」;「哈末多珥」被切成「哈末」+「多珥」——**全部只靠新增完整詞條就自動修好,沒動 `_EXT_GUARD`**。
- `_EXT_GUARD` 只在「短詞已註冊、但長詞本身不想/不能註冊成詞條」時才需要用(例如純方位詞組合)。判斷順序:先問「這個長詞本身值不值得成為一個詞條」,值得就直接加詞條;不值得才考慮加 guard。
- 另一種假陽性:短詞條字面剛好跟常見片語重疊,例如「以南」(民數記人物 Enan)在其他書卷幾乎都是「XX**以南**」的方位詞,被誤標成人名。這種不是碎片問題,是**詞條本身該加 `books` 限定**,已修正為 `books:["民數記"]`。
- `_SINGLE_CHAR_OK`:單字白名單(如 `噩`、`揝`、`圭`、`纛`)。單字預設不標,除非全經搜尋確認**唯一用法、零碰撞風險**才加白名單。

### JSON 編輯注意事項
用 Python script(`json.load` → 改 dict → `json.dump(..., ensure_ascii=False, indent=1)`)整批寫入,比手動 Edit 逐條改 `},` 括號安全很多,本輪全程用這個方式,沒再踩過括號漏改的坑。改完照樣要跑:
```bash
python3 -c "import json;json.load(open('data/entities.json'));print('JSON valid')"
python3 -c "import mark_bible;print(len(mark_bible.ENTITIES))"  # 順便確認模組能載入
```

### archaic 詞彙也可以標(非人名地名)
除了人名/地名,`type:"concept"` 也可以拿來標「現代讀者看不懂的古語詞」,例如本輪新增的「大解」(如廁的古語)、「驢腮骨」、「窗櫺」、「枳棘」。生僻字可以加 `"reading":"注音"` 欄位(參考既有的「帷子」「嗉子」等詞條格式)。但**常見詞不要濫標**——例如「網羅」在現代中文仍是常用詞(如「自投羅網」),即使讀者回報說沒標,也判斷為不需要標,這種取捨要靠自己判斷,不是照單全收。
