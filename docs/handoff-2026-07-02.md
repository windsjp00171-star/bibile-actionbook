## 專案是什麼

`bibile-actionbook` — 互動聖經閱讀網站（Flask + Supabase）。核心賣點是 **context-aware 標註引擎**：經文裡的人名/地名/族名可以點，跳卡片看背景、地圖、世系圖。

**核心原則（必讀，所有判斷都以此為準）**：
- 標註是為了讓大家讀得順暢，不是全文注釋。只標「真正生僻、會卡住讀者」的專有名詞。
- **準確永遠優先於數量**。不確定的、位置不詳的族譜「誰生誰」，寧可跳過不標。
- 標註前**先看經卷定時代背景**，**再看上下文語境**，**再看前後字避免碎片誤標**。
- **這個專案更深一層的目的**：不是加功能炫技，是讓聖經文字從「平面、有隔閡感的古文」變回它原本該有的立體感、真實感——地點是真實地理、異象有具體畫面、律法有實際內容，不是模糊的古代名詞堆砌。這是使用者反覆強調的初衷，判斷「要不要加這個東西」時可以拿來當標準。

## 本輪（2026-07-02 這一整個 session）做了什麼

除了上面列的回報整批處理，這次還做了一輪**回頭改善既有詞條品質**的工程，起因是使用者反應「以色列：古代以色列人的居住地」這種循環定義的描述等於沒講。已建立的新標準：

### 1. 地點詞條補「現代地理位置」（PR #91，549個地點詞條全掃過一輪）
在描述最後補一句「（今...）」，讓讀者知道這個地方在現代地圖上大約對應哪裡。**358個補上了，193個 API 判斷位置存疑/只是泛稱區域的刻意不強加（寧缺勿錯）**。
- 套用前務必檢查原描述是不是已經提過現代地名了，不然會重複講兩次（本輪抓到10個這種案例，如「尼尼微」原本就寫了「今伊拉克摩蘇爾」）。

### 2. 抽象神學/末世類詞條補具體內容（PR #92、#93）
不要只講結論（「帶出審判」），要把實際內容列出來。示範案例是「七印」：原本只說「帶出審判與末世事件」，改成逐印列出白馬/紅馬/黑馬/灰馬分別代表什麼、第五至七印各自的異象。同標準修了耶和華、人子、律例、誡命四個核心神學詞。

**⚠️ 神學內容的紅線（使用者非常在意，強調了兩次）**：
- 只寫**各宗派共同承認的經文事實層面**（如出3:14「自有永有」、十誡具體內容），不涉及救恩論、教會論、洗禮方式、預定論這類容易踩到宗派分歧或異端邊界的議題。
- 如果真的遇到有神學觀點分歧的地方，用「有不同看法」帶過，**不要替讀者選邊站當成定論**。
- **絕對不要錄入異端邪說** — 使用者原話「這個是最重要的！！」，每次寫神學相關內容前都要想一下這條是不是主流各宗派共識。

### 3. 舊約律例典章制度類優先於新約（PR #94）
使用者明確說「新約詞條都還好，舊約的律例典章制度才是真的需要備注」——舊約對現代讀者距離感最大。本輪補了「逃城」（完全沒收錄，民35/書20出現十幾次，列出六座城名字）、「感謝祭」「甘心祭」（平安祭子類別，原文有出現卻沒收錄）、幫「贖罪祭」「贖愆祭」加了明確對照句（這兩個很容易搞混）。

### 4. 動植物/食物/器物類補「現代對照」（PR #95）
目的：讓完全沒接觸過的物件有個現代生活的錨點。**⚠️ 重要教訓**：Cerebras 草擬的33個候選裡，約2/3只是把詞條名字換句話重複講一次（如「沒藥：即今日的沒藥樹脂香料」等於沒講新資訊），**務必人工逐條核對，篩掉循環重複的，只套用真正換了不同詞彙、有實質幫助的**（本輪套用了7個：芫荽子=香菜種子、棕樹=棕櫚樹等）。

### 5. 還沒做完的方向
使用者提到「風土民情」——象徵性習俗動作（如撕裂衣服表哀慟、脫鞋作交易憑證）——這塊完全還沒開始查，是下一個可以做的方向，但**先確認使用者要不要繼續**，不要自己預設接著做。

## 網站功能修正（本輪）

### 路線動畫閃爍問題（花了好幾輪才真正解決，PR #81→#85→#86→#87→#88→#89，過程記錄如下避免重踩）
根本原因：鏡頭跟著移動時**縮放（setZoom）**會讓 Leaflet 重新排版地圖圖磚，高頻呼叫（原本每影格/16ms都呼叫）會讓圖磚來不及載入完就被打斷、看起來像閃爍甚至整片空白。
- 中間繞了幾個彎路：先想說只是縮放太頻繁（節流門檻、每段只切兩次）→ 沒完全解決 → 一度改成視角完全固定不動（犧牲掉「鏡頭跟著移動」的沉浸感，使用者不滿意）→ **最終正解**：縮放全程算一次固定不變，只做平移（panTo）跟隨，因為純平移 Leaflet 處理得很平順、只有縮放才會導致圖磚重排。
- 教訓：**改動後如果 commit message 說做了 A，一定要實際檢查程式碼裡真的有 A**，PR #87→#88 之間就發生過「commit message 說加了鏡頭跟隨，但實際上忘記在 step() 函式裡真的呼叫 panTo」的疏失。
- 另外修了「播放中途切換路線會整個凍結不動」的 bug（`selectRoute()` 沒有把舊路線的動畫狀態歸零，新增 `_stopAnim()`）。

### 地圖清單改側邊欄（PR #89）
路線圖的停靠點清單、本卷地圖的地點清單，都從「地圖下方」改成「地圖右側欄」（Google 地圖路線那種左右並排），避免面板高度不夠時清單被擠到看不見。本卷地圖點清單/圖標改成直接彈出 Leaflet popup 顯示簡介，不整個跳走離開地圖。

## 關鍵基礎設施

### `tools/draft_entities.py` — 多 provider 草稿工具（本輪新增）
支援 `--provider groq|gemini|cerebras` 三選一（各有免費層，額度有限，用完換下一個）：
```bash
export GROQ_API_KEY=...      # 或
export GEMINI_API_KEY=...    # 或
export CEREBRAS_API_KEY=...
python tools/draft_entities.py 約書亞記 19 --provider gemini
```
**⚠️ 三個 key 都是使用者本輪即時提供的，沒有存在 repo 或任何檔案裡（正確做法，不要把 key 寫進 commit 或 docs），下次要用得再跟使用者要一次。**
Gemini 免費層每分鐘 5 次請求（RPM），會撞 429/503，遇到就換下一個 provider，不要死等重試太多次。用法教訓：不要讓 API 自己「開放式猜測」整章漏了什麼（雜訊很多），**把已知的明確詞/現代地理判斷這種「窄範圍、有標準答案」的任務丟給它**效果好很多。

### `.claude/settings.json`（本輪新增）
```json
{"model": "sonnet"}
```
預設模型設為 Sonnet，避免每次開新 session 忘記切換、誤用昂貴的 Opus 做量體工作。使用者原則：**Sonnet 做量大的機械性工作（回報整批處理），Opus 只在真正需要多層判斷的少數個案才手動開**。

### Git 工作流程（⚠️ 本輪踩過的坑，一定要知道）
使用者要求**修完測過沒問題就直接 merge，不用問**。squash merge 會產生新的合併 commit，跟本地分支的舊 commit 歷史不連續，**每次 merge 完，下一個 commit 前一定要先 `git fetch origin main -q && git checkout -B claude/bible-entity-processing-7ma6po origin/main -q` 重建分支**，不然下一次 push 會變成假衝突（`mergeable_state: dirty`），要花力氣重建才能解決。已經踩過兩次坑才確立這個習慣，直接照做：

```bash
# 每次 PR 合併後，開始下一個修改前：
git fetch origin main -q
git checkout -B claude/bible-entity-processing-7ma6po origin/main -q
# ...改動、commit...
git push --force-with-lease -u origin claude/bible-entity-processing-7ma6po
# 開 PR（draft:false，因為使用者要求直接合併）、merge
```
Stop hook 會抱怨 commit 沒有簽名/committer email 不是 noreply@anthropic.com，**使用者明確表示不在意，看到直接略過即可，不用花時間處理**。

## 資料結構關鍵知識（維持不變，重申重點）

- 多義詞條改成 **list**，`resolve_entity()` 四層優先度：書卷+章節 > 書卷 > 約別 > 通用。
- **碎片誤標優先用「登記完整詞條」解決**，regex 最長匹配優先會自動修正，不要動 `_EXT_GUARD`（本輪西西拉/瑪挪亞就是這樣修的）。
- 單字詞預設不標（`_SINGLE_CHAR_OK` 白名單），除非全經搜尋確認**唯一用法、零碰撞風險**（本輪新增了「縋」）。有些常見單字（如「但」，跟連接詞「但是」同形）**判斷後刻意不加白名單**，這是對的判斷，不要之後又手癢加進去。
- JSON 編輯一律用 Python script（`json.load` → 改 dict → `json.dump(..., ensure_ascii=False, indent=1)`），改完照樣要跑：
