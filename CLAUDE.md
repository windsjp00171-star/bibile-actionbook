# CLAUDE.md

給所有在這個 repo 工作的 Claude session 看的**持久標準**。跟 `docs/handoff-*.md`
不同：那些是「這一輪做到哪了」的進度快照，會過時；這份是「不管哪一輪都要守住的
判斷原則」，除非使用者親口說要改，否則不要因為單次對話裡的臨時措辭就偏離。

## 專案是什麼

`bibile-actionbook` — 互動聖經閱讀網站（Flask + Supabase）。核心賣點是
**context-aware 標註引擎**：經文裡的人名/地名/族名可以點，跳卡片看背景、地圖、
世系圖。資料唯一來源是 `data/entities.json`（詞條）與 `data/relationships.json`
（人物關係／家族樹）。

**這個專案更深一層的目的**：不是加功能炫技，是讓聖經文字從「平面、有隔閡感的
古文」變回它原本該有的立體感、真實感——地點是真實地理、異象有具體畫面、律法
有實際內容，不是模糊的古代名詞堆砌。判斷「要不要加這個東西」都拿這個當標準。

## 底線原則（最重要，時刻守住）

1. **準確永遠優先於數量**。位置不詳、關係不確定的族譜「誰生誰」、含糊的同名
   人物歸屬，寧可跳過不標、不連，不要為了湊數硬做判斷。
2. **神學紅線（不可退讓）**：只寫**各宗派共同承認的經文事實層面**（如出3:14
   「自有永有」、十誡具體內容）。不涉及救恩論、教會論、洗禮方式、預定論這類
   容易踩到宗派分歧或異端邊界的議題；真的遇到分歧，用「有不同看法」帶過，不
   替讀者選邊站當成定論。**絕對不要錄入異端邪說**——每次寫神學相關內容前都要
   想一下這條是不是主流各宗派共識。
3. **具體，但不是百科式深挖**。備註要讓讀者「看得到畫面」，不是要窮盡神學奧秘
   或做人物傳記式的刻劃。止於此，剩下的留給讀經本身跟聖靈光照。
   - 反面案例（已修正過的真實錯誤）：「割禮：神與亞伯拉罕立約的記號」——完全
     沒講割禮具體是什麼動作（割除包皮），等於沒回答問題。「舍客勒：古代重量
     貨幣單位」——連公克數都沒給，其他度量衡詞條都有。「以色列人：以色列的
     後裔，猶太人的祖先」——純循環定義。**寫完一個詞條，自問「如果讀者真的
     問『這個到底是什麼』，我這段話有正面回答嗎？」**
4. **人物詞條深度看重要性分級**：主要人物（大衛、摩西、保羅這種）可以多寫一點
   背景/事件；次要、只出現一兩次的人物（族譜裡的某某之子、保羅同工名單裡的
   一個名字）簡短交代清楚「是誰、跟誰有關係、做了什麼」就好，不用展開成小傳。
   但**不能因為次要就乾脆不收錄或寫得含糊到沒有實質內容**——使用者靈修使徒
   行傳時最大的痛點就是這些次要門徒被完全帶過。
5. **系譜／人物關係資料寧缺勿錯**：見下方「同名衝突」技術限制，連不安全的就
   不連，不要為了畫面好看硬湊家族樹。

## 內容撰寫的技術眉角

### 標註引擎是字面比對，不是語意理解
`mark_bible.py` 的 `annotate()` 用 entities.json 的 key 做 regex 字面比對，**entity
的鍵必須是經文裡真的逐字出現的字串**，不然永遠標不到、卻不會報錯（靜默失效）。
新增詞條前務必用 `python3 tools/bible_query.py verse <書卷> <章> <節>` 核對原文
用字，不要憑印象猜測措辭。加完一定要用
`python3 tools/bible_query.py mark <書卷> <章> --verses` 或
`python3 -c "import mark_bible as mb; print(mb.entity_card('詞', book='...', chapter=N))"`
逐一核對真的標到、標對地方。

### 同名詞條的消歧義與「同名衝突」限制
多義詞條用 list，四層優先度：書卷+章節 > 書卷 > 約別 > 通用（細節見
`mark_bible.py` 開頭註解）。**`relationships.json` 是用純名字當 key，不分身份**，
但 `entities.json` 有些名字底下是好幾個不同人的清單。`app.py` 的
`chapter_lineage()`／`owns_relationships()` 用「這個名字在本章解析出的是不是
清單第一筆（v[0]）」判斷能不能當家族樹的種子；被其他人當子女／父母目標連過去
時，卡片顯示走的是 `_entity_brief()`，直接抓 `v[0]`、不看上下文。
**結論：只把 relationships.json 的邊連到「單一詞條」或「清單裡剛好是 v[0]」的
身份，其他同名衝突且不是預設身份的，寧可不連**，避免使用者點開卡片看到不相干
的人。（已有 `rel_owner: true` 這個逃生門欄位可以讓非 v[0] 的清單項目也被
`chapter_lineage` 接受當種子，但整份資料目前完全沒用過——真的要用之前，記得
relationships.json 本身還是只有一份 flat 資料，同名兩人若都需要關係資料，這個
機制救不了，此時就是「寧缺勿錯」跳過的情境。）

### JSON 編輯流程
一律用 Python script（`json.load` → 改 dict → `json.dump(..., ensure_ascii=False,
indent=1)`），改完照樣要跑：
```bash
python3 -c "import json;json.load(open('data/entities.json'));json.load(open('data/relationships.json'));print('valid')"
python3 -c "import mark_bible;print(len(mark_bible.ENTITIES))"
python3 tools/bible_query.py routecheck   # 確認沒有新增的路線警示
```

### 碎片誤標
優先用「登記完整詞條」解決（regex 最長匹配優先會自動修正），不要動 `_EXT_GUARD`。
單字詞預設不標（`_SINGLE_CHAR_OK` 白名單），除非全經搜尋確認唯一用法、零碰撞
風險。

## Git 工作流程

squash merge 會讓本地分支跟遠端產生「假分岔」。**每次 PR 合併後，開始下一個
改動前一定要先重建分支**：
```bash
git fetch origin main -q
git checkout -B <目前工作分支> origin/main -q
# ...改動、commit...
git push --force-with-lease -u origin <目前工作分支>
```
Stop hook 會抱怨 commit 沒簽名／committer email 不是 noreply@anthropic.com——
使用者明確表示不在意，看到直接略過，不用處理。

PR 建立時預設開 draft；內容測過（JSON 驗證 + 標註逐一核對）沒問題、也沒有
review comment 要處理的話，可以直接轉 ready + merge，不用每次都問使用者
（這是使用者明確要求的既有共識）。

## 使用者風格與期望

- **對 token 成本敏感**：能省則省，驗證不要窮舉、描述不要寫得比必要的長。
- **喜歡「加減平衡」**：加新功能前想清楚是否也該做減法。
- **不喜歡被一直問「要不要繼續」**，但遇到真的是他該做決定的模糊地帶（範圍
  多大、要不要動架構/改程式邏輯）還是要問，不要自己貿然開工。
- 支持信仰核心，讓文字從平面死板變得立體生動是初衷，神學正統性是不可退讓
  的紅線（見上）。
- 目前還在自己逐卷讀經、有疑問才回報，不是要求主動整卷整卷去掃書卷或搶進度；
  抓到品質問題（像上面舉的循環定義例子）可以主動修，但大規模掃描新方向前
  先確認範圍。
- 影片/GIF 讀不了，只能看截圖或文字描述。

## 交接文件

CLAUDE.md 不需要每次更新進度，只在**判斷標準或技術眉角本身有變化**時才動它。
